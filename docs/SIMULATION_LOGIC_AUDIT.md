# Simulation Logic Audit — comprehensive review

_Produced by a 12-analyst + synthesis multi-agent pass over the whole simulation, judged against the DM/worldbuilder thesis. Findings verified against source._

## Cohesion verdict

7/10. The architecture is cohesive and correctly aimed at the DM audience: one conserved-ledger substrate, parallel lenses that read it rather than re-deriving it, explainable receipts everywhere, a bounded deterministic loop, and tests that assert direction-of-causality rather than brittle magnitudes. The shipped ledger refactor genuinely closed the famine/legitimacy dead-reads in the substrate, capacity, and system-state layers (verified: all five ledgers exist and causalState reads each once) — that work is real and earns most of the score. It is held back from 8-9 by the fact that the SAME dead-read/parallel-model bug class the refactor targeted still survives at the seams the refactor didn't reach: the organic-condition feedback misroute (the loop's headline feature), ~4 orphan signals computed-but-never-consumed, ~4 producer-less archetypes whose consumer trees are dead, several vocabulary mismatches dropping content into fallbacks, and two parallel power/food models that never reconcile. None of these throw; they quietly mislead, which for a tool whose entire value proposition is trustworthy 'why' receipts is the most corrosive failure mode. The corruption loop's stability is asserted only by unit tests on pure functions, never by the wired soak guard that documents itself as the guard. Fix the top ~6 wiring gaps and apply a 'wire-it-or-delete-it' sweep to every computed-but-unread signal and this is a 9: the design is already there, the debt is duplication and dead channels, not bad logic.

## Overall assessment

This is a well-architected DM/worldbuilder cohesion engine, not a hard-economy sim, and at that altitude its bones are strong: a 14-variable causal substrate with a neutral-50 default and signed, human-readable contributor traces; five conserved ledgers (food/governance/defense/healing/magic) that are read once and shared by the substrate, the capacity lens, and the system-state dials; a genuinely closed world-pulse feedback loop (pressure -> candidate -> roll -> apply -> condition -> re-derive) that is deterministic-yet-seed-varied, fully explainable via roll receipts, and bounded by mean-reversion + expiry + a soak guard. The conserved-ledger refactor demonstrably did its job in the consumers it touched (causalState, capacityModel, deriveSystemState all band off the same deficitPct/surplusPct/legitimacyScore, with regression tests pinning the formerly-dead reads). ## Cross-cutting themes

### Dead/mis-wired feedback channels — the ledger-refactor bug class, surviving at the seams
_world-pulse-loop, system-state-display, defense-threat, npc-system, region-realm-magic, causal-substrate_

The refactor's thesis was 'one conserved quantity, read once, no parallel model, no dead read.' It succeeded inside the substrate/capacity/system-state core but the same failure recurs wherever a signal crosses a module boundary. The flagship case: organic pressure-conditions hard-code affectedSystems to legitimacy+trade for every kind (candidateEvents.js:72), so an emergent famine/plague/crime condition cannot reinforce its own substrate variable and the 'why' receipt names the wrong systems — the loop's single most important organic feedback is broken. The resilience dial's dead prosperity strings, the dead 'embattled'/'high'/'low' threat branches, the npcProfile category mismatch, and the magicProfile 'pervasive' band that getMagicLevel never emits are all the same shape: a consumer keyed on a vocabulary the producer never speaks, silently routing to a fallback or zero.

### Orphan signals — computed, persisted, and tagged, but read by nothing
_conditions-chains, food-economy, region-realm-magic, causal-substrate_

Multiple subsystems pay to compute and carry a signal that no downstream consumer ever reads, so the work is invisible and (worse) some of these signals appear in DM-facing prose implying a consequence that never lands. merchant_wealth is in affectedSystems for 8 archetypes and emitted by the regional engine but is not one of the 14 substrate variables, so that pressure evaporates. foodLedger.magicSupplement and importDependency are billed as first-class conserved quantities but no consumer reads them (and magicSupplement is a SECOND magic calc that disagrees with the magicOffset that actually closes the deficit). MagicProfile computes legality/cost/risk/roles and is surfaced nowhere outside its own test. The honest move for this audience is wire-it-or-delete-it: every computed signal should either reach a DM-facing lens or be demoted to an internal local.

### Producer-less consequences — rich consumer trees with no event that ever fires them
_conditions-chains, events-mutate, corruption_

Several archetypes have full templates and many consumers (capacityModel, causalState, dailyLife, districtProfile, threatProfile) but NO code path ever creates them as a live condition: food_anchor_lost, corruption_exposed, siege_lifted. The corruption EXPOSE path that SHOULD raise corruption_exposed never calls withActiveCondition (verified mutate.js:493-543), and three registry events (STARTED_RIOT, REMOVED_THREAT, RECOVERED_RESOURCE) have no mutate handler so their effects vanish on re-derivation. The net DM experience is 'the scandal/famine-anchor-loss/siege-relief I expected to bite never shows up in the numbers.' These are the gameable beats a table most wants, and they are dead.

### Two parallel models for one fact that never reconcile
_factions, food-economy, npc-system, defense-threat, corruption_

Repeatedly, two independent representations of the same fiction coexist without write-back, so the sim can report a maneuver the headline number never reflects. Faction power: roster 0-100 vs factionCompetition 0-1, neither writes back. Food: the canonical foodGenerator/foodLedger vs a full duplicate caloric model in economicGenerator.buildFactionList with different defaults (road vs isolated) and different magic gates that drives the DM-facing viability warnings. NPC classification: dossier category-string vs world-pulse role-text archetype, which can disagree on the same NPC. Monster threat: config.monsterThreat (ambient) vs the monster_pressure stressor (acute), neither derived from the other. NPC relationships: a static generated graph vs world-pulse rivalry/ousting that never flows back. Corruption/ousting computed by world-pulse is never reflected into the dossier the DM reads.

### Inert lifecycle — labels and statuses that never actually move
_conditions-chains, capacity-model, npc-system_

Several fields advertise dynamics they don't deliver. Condition status 'worsening'/'easing' is purely cosmetic: severity is frozen for the condition's whole life, so a 'worsening plague' taxes the substrate by a flat amount for 12 ticks then vanishes in a single cliff at expiry — the opposite of compounding consequence. capacityModel trajectory is hard-coded 'stable' for every capacity, so a DM reading 'healing: strained, stable' during an escalating plague gets a contradictory cue. cold_war_sanctions has no catalog template so it never expires (null expiry) and sticks forever even after the relationship thaws. NPC goal-culmination requires long-progress >=0.8 that mean-reversion + context-rebranch resetting progress to zero makes structurally near-unreachable in a normal campaign.

### Vocabulary collisions surfaced to the DM in one panel
_system-state-display, causal-substrate_

The display layer bands on Stable/Strained/Vulnerable/Critical at 25/50/75 while the causal substrate bands on surplus/adequate/strained/critical/collapsed at 15/30/50/75. Both vocabularies contain 'Strained' and 'Critical' at incompatible thresholds, and the event pipeline concatenates BOTH delta lists into one 'what just happened' panel — so 'Resilience fell to Strained' (score ~30-50) sits next to 'Food security fell to strained' (score ~30-50 but a different range) meaning two different things. Resilience and causal food_security also read the same ledger but band it on different cut-points. For a tool whose value is legibility, one word meaning two scales side-by-side is a direct readability trap.

### Owner-declared-noise capacities still leak into DM-facing surfaces
_capacity-model_

Per owner decision, craft/welfare/transport/labor are noise for this audience and no ledgers should be built for them — honored. But as shipped they are NOT hidden: dailyLife.deriveDawnWork emits player-facing prose off labor and craft bands, deriveOutsiderImpressions lists ALL strained capacities (including craft/labor/transport) to outsiders, and all four appear in the aiGrounding bands fed to the AI. So there is a stated intent to suppress with full visibility in practice. This is a one-line visibility decision (gate DM-facing iteration to a CANONICAL_VISIBLE subset, OR deliberately reclassify labor+craft as flavor capacities) — explicitly NOT a recommendation to build any ledger.

### capacity-model

The capacity model is a supply-vs-demand lens over 9 settlement capacities, with the explicit purpose (header lines 17-26) of surfacing WHY a capacity moved (supply fell vs demand rose) and of representing events that push two capacities in opposite directions. As a DM cohesion tool the architecture is sound: each capacity emits signed supply/demand contributors with prose reasons (good "receipts"), the five conserved ledgers (food/defense/governance/magic/healing) are read once as supply inputs with deliberate de-dup of the old double-counts, and the trade-route semantics fix is correctly threaded. The band thresholds are coherent and the surplus/critical contradiction detectors give a DM real gameable tensions. However there are several concrete defects: (1) the flagship "refugee adds labor supply" example the module advertises is never implemented; (2) transport demand/supply is frozen to config and ignores every live condition/threat, so it can never move mid-campaign — yet it still feeds trade_connectivity prose; (3) explanation.js inverts the causality, telling the DM each capacity "feeds the substrate" when in fact both read the same ledger in parallel; (4) the band baselines mean many signal-poor capacities default to "strained" rather than the "adequate" the header claims, which over-reports crisis. On the owner decision: craft and labor both leak into DM-facing daily-life prose and labor into contradiction detection, so they are NOT pure noise as-shipped; transport and religious_welfare/welfare-style surfacing is thinner. Concrete recommendations below.

  The module header sells the whole design on an asymmetric event: 'Refugee influx raises FOOD DEMAND but adds LABOR SUPPLY. ### system-state-display

The four-dimension display layer (resilience / volatility / externalThreat / resourcePressure in deriveSystemState.js, diffed by compareSystemState.js) is well-conceived for the DM audience: four headline dials with drivers/risks and honest band labels, fed by the same conserved ledgers (food, governance) the causal substrate uses, so the recent ledger refactor genuinely closed the dead-read famine/legitimacy bugs HERE too. Within itself the layer is coherent and never throws. The real coherence problems are at the seams: (1) a dead prosperity-vocabulary branch means the three most common prosperity tiers (Poor/Moderate/Comfortable) contribute ZERO to resilience and one risk string ('Subsistence') can never fire — the headline silently ignores the most common economic state; (2) the display's four-band vocabulary (Stable/Strained/Vulnerable/Critical at 25/50/75) collides head-on with the causal substrate's five-band vocabulary (surplus/adequate/strained/critical/collapsed at 15/30/50/75), and the event pipeline shows BOTH delta lists to the same DM in one concatenated panel, so 'strained' means two different score ranges side by side; (3) resilience and the causal food_security read the same foodLedger but band it on different cut-points and resilience omits legitimacy/defense/healing entirely, so the headline 'resilience' and the substrate can tell different stories about the same town. None of these throw; they quietly mislead. For this audience the fixes are: align the prosperity strings, and either reconcile the two band vocabularies or stop showing them together.

  deriveResilience checks prosperity against 'Wealthy'/'Prosperous' (+15), 'Subsistence'/'Struggling' (-15), and 'Modest' (driver only). So: 'Subsistence' never appears (it is transformed to Struggling/Poor before storage) — that -15 risk branch is dead. 'Modest' is not in the vocabulary at all (the word is 'Moderate') — that driver is dead. And 'Poor', 'Moderate', 'Comfortable' — three of six tiers and the statistically common middle — match NOTHING, so they contribute 0 to resilience and produce no driver/risk. The sample dossier (sampleDossier.json:76) carries prosperity {tier:'Comfortable'} and gets zero resilience signal from its economy. A DM looking at the resilience dial and its 'why' list gets no economic explanation for the most common towns. ### world-pulse-loop

The loop is fundamentally sound and well-shaped for a DM cohesion engine: pressure -> candidate -> roll -> apply -> condition-on-settlement -> next-tick causal re-derivation is a genuine closed feedback loop, it is deterministic-yet-seed-varied via forked PRNG, every roll emits an explainable receipt (rollExplanations with gates/probability/roll), and decay (condition expiry + state mean-reversion + the soak guard) keeps it bounded. The volatility dial and intensity multiplier are clean, monotonic, and back-compatible. The single most consequential defect is a dead/mis-routed feedback path: organic pressure-conditions hard-code affectedSystems to ['public_legitimacy','trade_connectivity'] for EVERY pressure kind, which overrides the correct per-archetype routing in the condition catalog. The result is exactly the dead-read / parallel-model class the conserved-ledger refactor was meant to kill: an emergent famine does not lower food_security, an emergent plague does not lower healing_capacity, an emergent crime condition does not raise criminal_opportunity, so most organic conditions cannot reinforce or relieve their own substrate and a DM's "why is the famine getting worse" receipt points at the wrong systems. Secondary issues: a soft double-count between the organic-condition and stressor-birth pathways for the same pressure, and a couple of cosmetic dead fallbacks. Strengths clearly outweigh the defects, but the affectedSystems misroute should be the top fix.

### corruption

The damped corruption loop is well-conceived for a DM cohesion engine: it is gated (criminal institution required, so no-criminal towns stay byte-identical), hazard-driven with sensible signs, and genuinely bounded by three independent dampers (saturating guild strength, a bounded security drag, and the asymmetric "scar" that makes re-corruption harder while re-exposure gets easier). It produces gameable receipts — exposure events name the tied criminal + home institution, demote standing, oust+replace with a fresh successor, drag the faction up/down the criminalCaptureState ladder, and stamp thievesGuildStrength on the dossier. The math holds: probabilities clamp, the guild curve saturates, security never reaches zero. The most serious problems are not runaway risk but (1) the documented runaway guard does NOT actually exercise the corruption loop — the soak fixtures have no criminal institution, so every corruption/capture/guild path short-circuits on the very first line, leaving the loop's stability unverified by the test that memory/corruption-system.md claims guards it; (2) a dead-code parallel-model smell that the recent conserved-ledger refactor was specifically meant to eliminate — demoteImportance/canBeOuted are exported, unit-tested, and documented as the standing-erosion mechanism, but the live tick gates entirely on dotRank and never touches importance, so a 'pillar' NPC can be quietly outed after two demotions while the dossier still labels them a pillar; and (3) a vector that can never fire (fanaticism) plus tick-onset NPCs that lose their criminal-institution tie. Several DM-facing consequences a table would expect are also missing (no corruption surfaced as visible crime/safety pressure, no spoils/wealth signal, no PC bribe/leverage hook, capture transitions produce no narrative beat).

### npc-system

The NPC system is two loosely-coupled halves. (1) A generation + dossier layer (npcGenerator.js -> npcStructure.js -> npcProfile.js) that produces richly-textured static NPCs: names, personality, secrets, goals, faction seats, structural position, and a "what happens if removed" forecast. (2) A world-pulse agency layer (npcAgency.js + factionCompetition.js) that turns NPCs into ticking agents with role archetypes, goals, momentum/heat scalars, corruption onset/exposure, rivalry, goal-culmination payoffs, and faction-seat wiring. Individually each half is coherent and genuinely gameable for a DM — the action families, removal consequences, structural-tension templates, and goal-culmination arcs all answer real table questions. The serious problems are at the SEAMS: the two halves classify the same NPC by two different schemes (category-string vs. role-text archetype) that can disagree; a category-vocabulary mismatch silently routes generated NPCs (crafts, magic, noble) into wrong/fallback templates in the dossier; corruption and ousting computed by world-pulse are never reflected back into the dossier-facing profile a DM reads; and the static profile's leverage/vulnerability/consequence text is purely archetype-templated, ignoring the NPC's actual generated secret, relationships, faction power, and corruption state. For the DM audience the net effect is that the live, evolving NPC story (who got corrupt, who rose, who was ousted) and the static dossier the DM reads at the table can tell contradictory stories about the same person.
