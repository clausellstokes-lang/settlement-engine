# SettlementForge — The A+ Program

> **Authored 2026-06-16.** An exhaustive, prescriptive path to take every subsystem to A+, derived from the
> fresh-eyes evaluation (see [`RISK_REGISTER.md`](./RISK_REGISTER.md) for current grades + open risks and
> [`PRODUCT_COHERENCE.md`](./PRODUCT_COHERENCE.md) for the UX model). Every step names the concrete change
> (`file:line`), the **rationale** (the principle), and the **acceptance** test/pin/lint that proves it — because
> the central lesson of the review is that *a standard without an enforcing check is just a comment*.

**The one idea behind everything here:** this codebase already has world-class prevention *infrastructure*
(structural pins, the gate, determinism enforcement, privacy-by-construction). Its grade gap is almost entirely
the distance between what that infrastructure *could* enforce and what it *currently* does. A+ is reached not by
building more, but by closing that gap — making every claim the code makes about itself mechanically true, and
making every external seam fail visibly instead of silently.

## Reconciliation with the "path to 10/10" critique

A second external review proposed an 18-point path to 10/10. Read against the **actual code**, it
is highly complementary to this program but was partly written **without full repo knowledge** —
roughly a third of its asks are already built. The two reviews divide cleanly and completely:

- **The 10/10 critique supplies the *destination architecture + product/process layers*:** a North
  Star (typed commands → causal receipts → durable sync → contract-tested boundaries → shared view
  models), plus simulation eval fixtures, product E2E, performance budgets, a merge-gate operating
  standard, and second-engineer docs.
- **This A+ program supplies the *enforcement discipline, the verified current wounds, and the
  sequencing/pins that stop gains from rotting*.**

**Combined truth:** the North Star is the *target*; the enforcement spine is *how you reach it
without it silently regressing* — and the real combined backlog is **smaller than 18 items**,
because ~7 are already done and the rest mostly fold into the tracks below.

### North Star (adopted)
> *A deterministic settlement-simulation platform where every user action becomes a typed command,
> every command emits causal receipts, every mutation persists through a durable sync path, every
> public/private boundary is contract-tested, and every major UI/export surface renders from shared
> view models.*

This program's six principles are the **mechanism** for that vision: *one source per fact* → shared
view models; *fail-visible never fail-silent* → durable sync; *the gate is the product* + *claims
carry their enforcement* → contract-tested boundaries. Crucially — both reviews agree **do not
rewrite**; the architecture below is reached *incrementally, behind pins*, never as a big-bang CQRS
port.

## Governing principles (internalize these and the grades follow)

- **FAIL-VISIBLE, NEVER FAIL-SILENT** — every external seam (cloud save, analytics flush, provider fetch, webhook) must report failure into observable state — never console.warn-and-drop, never no-op early-return, never 'saved' while Supabase drifted. The persistSaveUpdate unify and the analyticsQueue force-drain are the two flagship instances of this single rule.
- **ONE SOURCE PER FACT** — any value shown to a user must be derived once and consumed everywhere — the food-deficit-% (deriveFoodBalance), the institution classification (catalog-derived), the tag vocabulary (entityTags.TAG), the persist helper (single impl). Divergent re-derivation is the recurring defect generator across PDF, generators, and data plans.
- **PURE BY CONSTRUCTION, NOT BY DISCIPLINE** — non-determinism and IO leaks must be made structurally impossible (widened determinism eslint guard over domain, directory-scoped purity lint on src/data, no flag/Math.random/new Date in the kernel) so the property holds under refactor, not just under reviewer attention.
- **THE GATE IS THE PRODUCT** — CI must be the only path to production and must EXECUTE the real artifact for every money/RLS boundary (signed-vs-forged webhook, pglite RLS, refund stream) — grep-over-source is decoration. No green `check`, no deploy.
- **CLAIMS MUST CARRY THEIR ENFORCEMENT** — the meta-pin makes every documentation completeness claim name a resolvable @enforced-by test/lint, so the gap between 'we claim X' and 'the gate proves X' can never silently reopen. This is the theme that keeps all the others from regressing.
- **STRUCTURE LOCKED BY PINS** — every decomposition (settlementSlice sub-slices, god-component splits, generator step extraction) is fenced by a structural pin (max-lines, single-impl grep, contract test, golden master) so the improvement cannot be silently undone — byte-identical same-seed output is the non-negotiable invariant for engine refactors.

**A+ bar.** Zero raw PII in any edge-function log line, governed by a single shared structured-logger that scrubs emails/IPs by policy (not by per-call discipline); every money-path and authz boundary proven by an EXECUTED test (function actually invoked, not source-text regex), wired into `npm run check`; and a CI guard that fails the build if a future diff reintroduces a raw-PII log. No remaining unpinned SECURITY DEFINER, no RPC reachable by anon/authenticated that should be service-role-only.

**Highest leverage.** Scrub the confirmed plaintext-email leak at stripe-webhook/index.ts:255 (and the error-object leak at :85), then make it structurally impossible to recur by routing all edge-function logging through one shared `_shared/log.ts` helper that redacts emails/IPs — and lock it with a CI grep guard. This kills the one verified finding and its entire bug class in a single move.

**Sequencing.** Do Step 1 first (it is the only confirmed defect and is trivially safe). Step 2 (shared logger) should land with Step 1 so the fix is structural, not a one-off. Step 3 (CI grep guard) depends on Steps 1-2 being done or it fails red immediately — land it last in the same PR. Step 4 (execution tests) is the largest item and the real A+ unlock; it is independent of 1-3 but MUST coordinate with the testing area to avoid two parallel deno-test harnesses — agree on one runner (`deno test` invoked from a vitest shim or a new `test:edge:exec` script) before writing cases. Step 5 (test-assertion fix at contracts.test.js:162) is a hard dependency of Step 1 — the existing test asserts `customer_email` presence and will go red the instant you remove the log; bundle it into Step 1's commit. Step 6 (identifier-logging policy) is polish, do after 1-5.

### backend.1 Scrub the plaintext customer_email from the stripe-webhook single_dossier log
`critical · effort S · low risk`

- **Change:** supabase/functions/stripe-webhook/index.ts:255 — change `console.log(`single_dossier purchased: session=${session.id} email=${session.customer_email}`)` to log only the non-PII correlation key: `console.log(`single_dossier purchased: session=${session.id}`)`. The Stripe session id alone lets audit join back to the payment (and the email) in the Stripe dashboard, so no investigative power is lost. Also fix the secondary leak at line 85: `console.warn('[stripe-webhook] customer lookup failed:', e)` — `e` is a Stripe error object whose message/raw can embed the queried email; replace with `console.warn('[stripe-webhook] customer lookup failed:', (e as Error)?.message ?? - **Why:** Supabase function logs are retained, exportable, and viewable by anyone with project log access — they are NOT a PII-appropriate sink under GDPR/CCPA. A pseudonymous correlation id (Stripe session id) preserves 100% of the audit trail without storing personal data at rest in logs. Principle: logs correlate, they do not store PII.
- **Done when:** `grep -rn 'customer_email\|\.email' supabase/functions/stripe-webhook/index.ts | grep console` returns nothing; `grep -n 'email=' supabase/functions/stripe-webhook/index.ts` returns nothing. Existing contract test (after Step 5 edit) passes: `npm run test -- contracts`.

### backend.2 Introduce a single shared structured logger (_shared/log.ts) that redacts by policy
`high · effort M · med risk`

- **Change:** Create supabase/functions/_shared/log.ts exporting `log.info/warn/error(fn: string, event: string, fields: Record<string,unknown>)` that emits ONE JSON line `{ts, level, fn, event, ...fields}` and runs every string field through a `redact()` pass: replace any `/[^\s@]+@[^\s@]+\.[^\s@]+/g` match with `'[email]'` and any value under a key matching `/email|ip|ua|token|secret|key/i` with a masked form (e.g. keep last-4 of an id, drop the rest). Then migrate the call sites: stripe-webhook (11 console.* calls), send-email (6), verify-single-dossier (3), admin-actions (1), requestMeta.ts (2). Replace freeform `console.log(`User ${userId} upgraded...`)` with `log.info('stripe-webhook','premium_upgrade',{userId})` etc. requestMeta.ts:119 `ua=${meta.ua.slice(0,200)}` and ip become structured fields the redactor governs centrally.
- **Why:** The current per-call-site discipline is exactly why a raw-email log shipped: correctness depends on every author remembering. Centralizing redaction makes the safe path the only path — a new field is scrubbed by default, and observability gets a queryable JSON shape (event names, levels) instead of grep-only prose. Principle: make the secure behavior structural, not a convention.
- **Done when:** All `console.` calls in supabase/functions/*/index.ts and _shared/requestMeta.ts route through log.ts (verify: `grep -rn 'console\.' supabase/functions --include='*.ts' | grep -v _shared/log.ts` returns only lines inside log.ts itself). A unit test `tests/edgeFunctions/log.redact.test.js` asserts `redact('paid by a@b.com')` contains `[email]` and not `a@b.com`. `npm run validate:edge` still green.
