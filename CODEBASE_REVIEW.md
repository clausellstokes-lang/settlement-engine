## Overall grade

**B+ — Excellent engineering discipline and a fundamentally sound money/security core, dragged down by a systemic pattern of correct-looking mechanisms that are never actually enforced or wired in, plus a few foundational determinism and deploy-gating gaps.**

## Executive summary

SettlementForge is a genuinely well-engineered codebase — far more disciplined than typical indie/hobby-scale SaaS. The deterministic simulation engine (worldPulse, generators, region propagation) is the architectural core, and it is built with seriousness: randomness is funneled through a small set of sanctioned seams (prng.js, rngContext.js, clock.js), RNG forks on stable content keys rather than list position, sorts deliberately use codepoint comparison over localeCompare where outcomes are decided, and "byte-identical-when-disabled" dormancy gating lets new features ship dark without perturbing existing replays. The server-side security posture is layered and clearly the product of multiple prior audit passes: the Stripe money path verifies signatures before any DB write, routes all grants through an idempotent service-role RPC, and is proven replay-safe by an in-process Postgres (pglite) execution suite — the single best test in the repo. The data/derivation layer leans hard on shared "ledgers" to cure the "two models disagree" class of bug, and the React map render layer correctly mutates the SVG transform directly to avoid per-frame re-renders.

The architecture's recurring strength is also its recurring liability: the team builds elaborate, correct-looking mechanisms whose enforcement or wiring is incomplete, so a feature appears to work while silently doing nothing or contradicting itself. The clearest critical example is the war/occupation layer: it mints war_front regional-graph channels on deploy but never retires them on siege resolution, so a resolved siege re-fires conquest every tick and the entire occupation state machine (stabilization, vassalization) is dead in production — masked by tests that rebuild the graph from a constant array instead of threading it forward. The same shape recurs everywhere: account ban/disable/soft-delete flags that nothing enforces (banned users keep full JWT access and can still spend credits), store actions wired to monetization moments that production never calls (first_save/third_save, lifetimeNarrateCount stuck at 0), goods-toggle penalties whose backing data field no longer exists, and ~450 lines of stale-duplicate helpers that have diverged from their live twins. These are not isolated nits — they are a systemic gap between intent and enforcement.

The biggest concrete risks are: (1) the dead-wiring class above, which produces silent correctness failures and false confidence; (2) determinism breaks in the custom-content path (localeCompare-before-RNG) that violate the product's foundational "same seed => same settlement" guarantee for non-ASCII names; (3) operational gating — production deploys on Vercel independently of CI, so a red money-path test cannot stop a live deploy, and the CSP that the current branch is named for ships Report-Only with no sink (zero XSS enforcement) for a payment app; and (4) performance scaling of the synchronous, main-thread world-pulse advance (O(E^2) relationship pass, ~9x snapshot rebuilds, ~4N deep clones per tick), which will jank the UI on large campaigns. None of these are crashes on the common path, but several distort output, leak privacy boundaries, or undercut the deploy/security gates that a Stripe-integrated product depends on.

## Cross-cutting themes

### T1. Elaborate mechanisms that are never actually enforced or wired in (dead wiring)

The dominant systemic risk. Correct-looking, well-commented machinery exists but production never invokes it or nothing enforces its output, so features appear to work while silently doing nothing or contradicting themselves — and comments actively assert the opposite. This is more dangerous than ordinary dead code because it produces false confidence at the trust boundary.

_Affected: worldPulse war/occupation (war_front never retired, occupation state machine dead); Server security (ban/disable/soft-delete flags enforced by nothing); Zustand store (saveSettlement, spendCredits, bumpLifetimeNarrate never called); Content data layer (goods-toggle institution penalty reads a field no data defines; noble/HISTORY_EVENTS buckets unreachable); Generators (structuralValidator getPriorityModifiers machinery dead)_

### T2. Tests that pass without exercising the real behavior (false-confidence coverage)

Multiple high-impact bugs survive a green test suite because the tests assert source-text shape, validate superseded migration bodies, rebuild persistent state from constants instead of threading it forward, or use conditional/vacuous assertions. The exact failure modes the tests exist to catch are invisible to them.

_Affected: worldPulse war/occupation (cross-tick graph rebuilt from const array); Testing (regex-over-source edge contracts; superseded gallery/refund migration contracts; vacuous e2e assertions; runIf vacuous-skip); Build/CI (CI not a required check; local pre-push gate omits Deno+e2e)_

### T3. Determinism is engineered but leaks at the seams

Core determinism is genuinely rigorous, which makes the residual leaks especially costly because the product's value (snapshots, replays, same-seed reproducibility, property tests) rests entirely on it. The leaks cluster around locale-dependent ordering/formatting and RNG-consuming validators run outside the seeded pipeline.

_Affected: Generators/domain determinism (localeCompare-before-RNG in custom-content path; toLocaleString in stored descriptions); worldPulse simulation core (same-tick same-id stressor last-write-wins); Core domain (structuralValidator embeds a raw RNG roll on the live draft path); Reconciliation/schema (reconciliationLog stamps wall-clock when options.now omitted)_

### T4. Numbers and narratives that disagree within one artifact

The codebase fights the 'two models disagree' bug well with shared ledgers, but incomplete migration to those ledgers and wrong-shape headline/slice calls let different surfaces report contradictory facts about the same settlement — within a single PDF or between PDF and screen.

_Affected: Core domain (deriveHealing/deriveSocialTrust read stale vocabulary; cross-surface threat double-count); PDF generation (Relationships headline always empty; Resources headline drops clauses; daily-life food line edge cases); Regional graph (three mutually-inconsistent legacy relationship maps; ally/overlord mapped to neutral)_

### T5. Trust boundary depends on undocumented load-bearing assumptions

Security is correctly designed as defense-in-depth with the server as sole authority, but several gates rely on an assumption that is true today and undocumented at the call site, so a future regression silently converts a UX-only gate into a real bypass.

_Affected: Client security (client 'elevated' role gates credit-free AI; OWNER_EMAIL in bundle); Server security (CORS falls open to '*' when env unset; fail-open rate limiters; ban flags unenforced); Store (isTierAllowed unsafe for sentinel tiers in isolation)_

### T6. Synchronous main-thread simulation scales superlinearly

The world-pulse advance compounds several superlinear passes inside one synchronous Zustand/Immer transaction, fine for a handful of settlements but a guaranteed UI-freeze and GC-pressure trap as campaigns grow.

_Affected: Performance (O(E^2) relationship pass; 9x snapshot rebuild; ~4N clones/tick; main-thread set()); Relationship dynamics (npcStates never pruned; O(F*N) faction/rivalry scans); Regional graph (regional graph re-normalized ~3N times/tick); UI (getAllModifiers O(N^2) over whole library on every saves change)_

## Full findings by area

### worldPulse conflict & lifecycle subsystems (war deployment, occupation, mobilization, attrition/reinforcement, deployment return, feasibility gate, institution lifecycle, tier/resource dynamics)

`🔴1  🟠1  🟡1  🔵2  ⚪1`

This is an unusually well-engineered simulation layer. The determinism contract (no wall-clock/Math.random, codepoint-sorted iteration everywhere, read-last/write-next ledgers, rng forked on stable keys, conditional ledger materialization for byte-identity) is taken seriously and consistently honored; the anti-snowball containment caps in occupation.js and the bounded attrition/reinforcement kernels are textbook. The state machines (mobilization ramp, occupation hysteresis dwell) are clean and the feasibility gate correctly sits as a deterministic classifier in front of the siege RNG. However, there is one CRITICAL architectural gap: the war layer mints `war_front` regional-graph channels on deploy but NEVER retires them when a siege resolves (conquest/withdrawal). Because the regional graph persists across ticks in production (returned as `applied.regionalGraph`) and `addRegionalChannels` keeps channel status sticky, a resolved siege leaves a phantom `confirmed` war_front that re-triggers the siege every subsequent tick — I reproduced 9 conquests across 12 ticks for a single deployment. The existing tests mask this because they rebuild the graph from a constant `channels` array each iteration instead of threading `pulse.regionalGraph` forward, so they never exercise channel persistence. This one bug cascades into the occupation layer (perpetual re-seed to `contested`, so occupations can never stabilize/vassalize), disposition ledger saturation, and chronicle/realm-event spam. A second, lower-severity correctness bug: stabilized `vassalized` occupations re-emit the vassalization relationship outcome every tick. The pure kernels (attrition, reinforcement, feasibilityGate, militaryStrength, mobilization, blockadeTransport) and the lifecycle modules (institutionLifecycle, tierResourceDynamics) are sound on their own.

**Strengths:** Rigorous, consistently-applied determinism: every output-affecting iteration is codepoint-sorted, rng is forked on stable keys (siege:T:tick, deploy:S:tick), and cross-settlement reads come from a single pre-tick snapshot with explicit read-last/write-next discipline (warDeployment.js:618-654, mobilization.js:381-434).; The anti-snowball occupier-benefit containment in occupation.js:404-439 is genuinely robust: per-occupation cap, diminishing returns BEFORE a hard per-occupier sum cap, with a soak test pinning the bound — exactly the right shape to prevent compounding cross-settlement feedback.; Clean separation of the deterministic feasibility classifier (feasibilityGate.js) in front of the stochastic siege roll, making 'RNG only resolves plausible conflicts' itself reproducible.; Conditional ledger materialization in worldState.js:65-69/136-194 (pantheon/warPosture/occupations stripped from the shallow spread and re-added only when non-empty) is a careful, well-documented solution to byte-identity under the dormancy oracle.; The attrition and reinforcement kernels (attrition.js, reinforcement.js) are pure, bounded (per-engagement loss cap, flow capped below the deficit and never above maxStartStrength), and well-calibrated with documented tunables.; Institution lifecycle damping (hysteresis streaks, saturation via prior-event penalties, clamped probability bands, history-cap amnesty) in institutionLifecycle.js is a thoughtful no-runaway design, and apply-time re-verification of closability guards against stale proposals.

### Cross-cutting correctness/determinism audit of src/generators and src/domain

`🔴0  🟠2  🟡2  🔵3  ⚪2`

The team has been impressively disciplined about determinism. There are zero live (non-comment) calls to Date.now(), new Date() (argless), performance.now(), or Math.random() anywhere in src/generators or src/domain outside the three sanctioned seams (prng.js, rngContext.js, clock.js). Wall-clock reads are funneled through clock.js and threaded as a `now` parameter into the simulation; trace timestamps use a deterministic monotonic ctx._traceClock counter. The contest engine (contestOverThirdParty.js + contestMath.js) is a model of careful determinism: content-keyed RNG fork recipes, FNV-1a/fmix32 integer hashing, numerically-stable softmax, hash-based (not alphabetical) tie-breaks, and codepoint-stable default sorts on settlement ids. The worldPulse trade-war/war/religion features explicitly codepoint-sort their iteration keys. The migration runner (settlementMigrations.js) is well-formed: linear chain, idempotent, gap detection, runaway-bound, plus a self-check (diagnoseMigrationChain). The single global mutable PRNG context (_activeRng) is set/cleared synchronously per pipeline step and is safe for the current single-threaded, non-nested generation flow.

The real determinism weaknesses cluster around two patterns. First, localeCompare is used as a "stable" sort key in the CUSTOM-CONTENT generation path (assembleInstitutions.js, resolveResources.js) immediately before per-item rng.chance() draws — but localeCompare collation is locale/ICU-dependent, so for non-ASCII custom names the RNG draw order (and thus which custom institutions/resources are selected) can differ across machines for the same seed, breaking the "same seed => same settlement" guarantee. The in-code comments literally claim this ordering "keeps rng deterministic," which is incorrect. Second, the food/economy plausibility warnings embed population.toLocaleString() (no locale arg) into stored description strings, and structuralValidator embeds a raw RNG roll inside what should be a pure validator — which produces flickering, machine-dependent results when the validator runs outside the seeded pipeline (the live draft-edit path). Neither corrupts core structural numbers, but both violate the snapshot/replay contract the product is built on.

**Strengths:** No live nondeterministic time/random calls anywhere in generators/domain outside the 3 sanctioned files (prng.js, rngContext.js, clock.js) — verified by grep across the whole scope.; contestMath.js/contestOverThirdParty.js use platform-stable integer hashing (Math.imul FNV-1a + fmix32 avalanche), numerically-stable shifted softmax, and hash-based tie-breaks — genuinely cross-platform deterministic and well-reasoned.; Determinism hazards are recognized and handled deliberately in outcome-deciding sorts: codepoint comparison (not localeCompare) is explicitly used where a sort picks a winner (stressorDynamics.js:612, applyWorldPulse.js:583, institutionLifecycle.js:46).; settlementMigrations.js is a clean, idempotent, gap-checked linear chain with a separate diagnoseMigrationChain() self-test and a runaway bound.; Trace timestamps are made deterministic via a monotonic ctx._traceClock counter rather than wall-clock, preserving snapshot stability.; idFromSeed/contentId in normalizeSettlement.js use deterministic integer hashing over fixed-key-order JSON, keeping ids replay-stable and the normalize pass idempotent.

### Client-side security: gallery XSS, auth/session handling, feature flags, analytics PII, and bundle secrets

`🔴0  🟠0  🟡0  🔵4  ⚪2`

The client security posture in scope is, overall, well-constructed and noticeably more careful than typical React SPAs. The gallery XSS surface — the one place untrusted user HTML is rendered — is genuinely well-contained: the only two `dangerouslySetInnerHTML` sites (GalleryCard.jsx:154, GalleryDetail.jsx:171) both pass through `sanitizeGalleryHtml()`, which wraps DOMPurify 3.4.8 with a tight tag/attr allowlist, a `https?:|mailto:`-only URI regexp, and an `afterSanitizeAttributes` hook forcing `rel="noopener noreferrer nofollow"` on anchors. Sanitization is applied defense-in-depth: on store (GalleryDescriptionEditor.onChange), and again on render. User comments render as React-escaped plain text, not HTML. No `eval`, `new Function`, or stray `innerHTML` writes exist outside the editor's own sanitized seed. Secrets are clean: only `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` (both client-safe) reach the bundle; `.env` is gitignored and `.env.example` correctly documents Stripe/Anthropic/Resend secrets as Supabase-dashboard-only. Session handling is thoughtful — the custom storage adapter routes "remember me off" tokens to sessionStorage and purges the opposing store to prevent token resurrection, and `profiles.role` (not user_metadata) is the trust source for privilege. The main residual concerns are not bypasses but trust-boundary clarity: a global DOMPurify hook that mutates all future `sanitize()` calls, a couple of fragile sanitizer config choices, and analytics props that flow to localStorage spill without a PII schema guard. The client correctly treats its own auth gates as cosmetic, but that assumption is load-bearing and undocumented at several call sites.

**Strengths:** Gallery XSS is correctly closed: both dangerouslySetInnerHTML sites (GalleryCard.jsx:154, GalleryDetail.jsx:171) sanitize via DOMPurify with a tight allowlist; sanitize runs both on-store and on-render (defense in depth); DOMPurify is the only HTML sink and is confined to one module.; URI scheme allowlist (sanitizeGalleryHtml.js:17) plus FORBID_TAGS/FORBID_ATTR for script/style/iframe/object/embed/form/input and style/class/id blocks the common javascript:/data: and CSS-exfiltration vectors; the createLink editor path is re-sanitized on emit so a javascript: link cannot persist.; No client bundle secrets: only Supabase URL + anon key are VITE-exposed; .env is gitignored and .env.example explicitly routes Stripe/Anthropic/Resend secrets to the Supabase dashboard.; Privilege resolution trusts profiles.role from the DB, never user_metadata (auth.js:17, fetchProfileAuth), and fails closed to non-privileged defaults on any profiles fetch error.; Session persistence adapter (supabase.js) genuinely fixes the 'remember me off' token-resurrection bug by purging the opposing store and routing auto-refresh writes through the adapter.; Consent + DNT gating is honored client-side as a hard pre-enqueue filter (consent.js, analytics.js:107/120), research-class data is dropped on revoke before flush (analyticsQueue.purgeRevoked), and third-party mirror only ever receives essential-class events with a SHA-256-hashed user id.; Auth error messages are deliberately non-leaky (describeOAuthError, supabaseReauthenticateWithPassword) to avoid account-enumeration.

- 🔵 **LOW · security** — Global DOMPurify.addHook mutates a process-wide singleton; any future DOMPurify caller silently inherits link-rewriting
  - **Where:** `src/lib/sanitizeGalleryHtml.js:20-30`
  - **Impact:** If a second feature later calls DOMPurify.sanitize for a different context (e.g. an admin tool that wants same-tab links, or a richer allowlist), it will silently inherit gallery's anchor rewriting and possibly a stale ALLOWED_URI assumption — a latent correctness/security footgun. Hooks also can't be scoped per-call, so the global mutation is invisible at the other call site.
  - **Fix:** Use an isolated instance: const purify = DOMPurify(window) (or DOMPurify.removeHook discipline), attach the hook to that instance only, and sanitize through it. Alternatively drop the hook entirely and rely on ADD_ATTR + a post-pass, or document loudly that DOMPurify is reserved exclusively for this module.  _(confidence: high)_
- 🔵 **LOW · security** — Sanitizer allowlists target/rel as attributes but relies solely on a hook to set rel — malformed/anchor-only output if the hook ever fails to register
  - **Where:** `src/lib/sanitizeGalleryHtml.js:15-29`
  - **Impact:** If DOMPurify is ever swapped/shimmed without addHook (or under an unusual build), user-authored links render with target=_blank and no rel, reintroducing reverse-tabnabbing (window.opener) on the public gallery.
